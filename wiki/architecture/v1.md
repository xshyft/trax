# TRAX Architecture v1

This page describes the current standalone architecture of TRAX after extraction from `daemons2`.

## Runtime surface

- **Go module:** `github.com/kamcpp/trax`
- **Daemons:** `traxctrl`, `traxcoord`
- **CLI:** `traxcli`
- **Core package:** `pkg/trax`
- **Store backends:** PostgreSQL and in-memory
- **Transport:** RabbitMQ
- **Cache/locking:** Redis-backed cache abstraction

## Main flow

1. A submitter announces to `traxcoord` and becomes ready.
2. A workflow instance is submitted into a cluster.
3. The coordinator validates against the stored template and persists saga/step instances.
4. The coordinator publishes the next runnable step request.
5. An executor consumes the request and publishes a step result.
6. The coordinator persists the result and either:
   - schedules the next step;
   - marks the workflow terminal;
   - or drives compensation.

## Extracted state

The standalone repo currently contains:

- `pkg/trax` and watch helpers;
- TRAX daemons and APIs;
- TRAX CLI and executor/submitter/template surfaces;
- PostgreSQL init SQL and namespace seed SQL;
- dedicated TRAX E2E tests and their framework.

## Legacy coupling still present

A few extracted packages still carry more breadth than TRAX itself needs:

- `pkg/common` includes legacy helpers from `daemons2` that should eventually be reduced.
- `go.mod` still includes many transitive dependencies that came over with the extraction.
- some docs and helper code still reference `agora_db` or older image naming conventions.

Those do not change the runtime ownership boundary, but they are part of the current state.
